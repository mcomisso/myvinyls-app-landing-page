# My Vinyls website

This repository contains the public marketing and editorial website for
[My Vinyls](https://myvinyls.app), an iPhone app for managing a vinyl record
collection with Discogs sync, barcode scanning, offline access, and
collector-focused insights.

The site is a customised Jekyll project hosted with GitHub Pages. It is not the
My Vinyls iOS app or a reusable landing-page template. The app source lives in
`/Users/matcom/Developer/MCSoftware/MyVinyl/iOS`.

## What is in this repository

- `index.html` — product landing page, feature overview, App Store calls to
  action, and Android waitlist
- `blog/` and `_posts/` — The Vinyl Journal index and articles
- `_pages/` — FAQ, support, about, contact, privacy, and changelog pages
- `_layouts/` and `_includes/` — shared Jekyll page structure and components
- `assets/` — styles, scripts, app screenshots, video, artwork, and press kit
- `_config.yml` — site metadata, App Store details, feature copy, theme values,
  blog configuration, and Jekyll settings
- `.well-known/apple-app-site-association` — Apple universal-link association
- `app-ads.txt`, `robots.txt`, `sitemap.xml`, and `llms.txt` — discovery and
  platform metadata (some files are generated by Jekyll)
- `get/` — lightweight App Store redirect page

The dynamic discovery site at `dig.myvinyls.app` is a separate project in
`../the-crate`. Backend APIs are maintained in `../vinyly_backend`.

## Run locally

The project uses Ruby and Bundler. Install the locked dependencies, then start
Jekyll with live reload:

```sh
bundle install
bundle exec jekyll serve --livereload
```

Open [http://localhost:4000](http://localhost:4000).

To create the same static output without running a server:

```sh
bundle exec jekyll build
```

The generated site is written to `_site/`. Do not edit generated files there;
make changes in the source pages, layouts, includes, Sass, or assets instead.

## Common updates

### Site and product copy

Edit `index.html` for landing-page sections. Shared product metadata and the
feature list live in `_config.yml`; restart the local Jekyll server after
changing configuration.

### Blog posts

Add Markdown files to `_posts/` using Jekyll's
`YYYY-MM-DD-descriptive-slug.md` naming convention. Existing posts are the best
reference for front matter, categories, images, and article structure. Blog
URLs use `/blog/:year/:month/:title/`.

### Support and legal pages

Edit the relevant source in `_pages/`, `privacy.md`, or `terms.md`. The FAQ uses
its own layout and interactive accordion markup.

### Images and video

Keep product screenshots in `assets/screens/`, editorial artwork in
`assets/blog/images/`, and landing-page media in `assets/media/`. Prefer
optimised web formats and include useful alternative text for content images.

## Validation

Before publishing, run:

```sh
bundle exec jekyll build
```

Review the build for warnings, then check the affected pages locally at both
desktop and mobile widths. For content changes, also verify page titles,
descriptions, internal links, image paths, and the generated sitemap/feed where
relevant.

## Deployment

The production domain is configured by `CNAME` as `myvinyls.app`. The `master`
branch is the publishing branch for the GitHub Pages repository. Pushing a
validated change to `master` triggers the configured Pages build and deploy.

## Project guidance

Read `AGENTS.md` before making changes. `PRODUCT.md` captures the site's users,
purpose, voice, and design principles. Keep this project focused on the public
marketing and editorial website; app features and backend business logic belong
in their owning repositories.
